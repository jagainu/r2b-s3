# 運用方針定義

## チーム構成

| 役割 | 担当者 | 人数 |
|------|--------|------|
| PM | まことPM | 1 |
| フロントエンド | - | 1 |
| バックエンド | - | 1 |
| インフラ（PM兼任） | まことPM | 1 |

**合計**: 3名

## GitHub ブランチ戦略

**採用戦略**: Git Flow（develop ブランチあり）

### ブランチ構成

| ブランチ | 役割 |
|---------|------|
| `main` | 本番リリース済みコード。直接コミット禁止 |
| `develop` | 開発統合ブランチ。feature をマージする先 |
| `feature/*` | 機能開発ブランチ。develop からブランチアウト |
| `hotfix/*` | 緊急修正ブランチ。main からブランチアウト |

### 命名規則

- feature ブランチ: `feature/{issue番号}-{機能名}`（例: `feature/123-quiz-screen`）
- hotfix ブランチ: `hotfix/{issue番号}-{修正内容}`
- コミットメッセージ: Conventional Commits（`feat:`, `fix:`, `docs:`, `refactor:` など）

### PR マージ条件

- レビュー承認: 1名以上
- CI チェック: 通過必須
- セルフマージ: 許容（レビュー1名条件を満たせば可）

## デプロイ・インフラ方針

**デプロイ**: あり
**デプロイ先**: AWS（ECS Fargate + RDS + S3 + CloudFront）

### 環境構成

| 環境 | 場所 | 用途 | 主要サービス |
|------|------|------|-------------|
| dev | ローカル（Docker） | 開発・動作確認 | Docker Compose（FastAPI + PostgreSQL） |
| stg | AWS | 本番前検証・レビュー | ECS Fargate / RDS / S3 / CloudFront |
| prod | AWS | 本番 | ECS Fargate / RDS / S3 / CloudFront |

### SLA と可用性

| 項目 | 設定値 |
|------|--------|
| 目標稼働率（SLA） | 99.9%（年間8.7時間ダウン許容） |
| フェイルオーバー | 自動（RDS Multi-AZ） |
| Auto Scaling | なし（固定スペックで運用・コスト最適化） |

### トラフィック予測

| 指標 | 見込み値 |
|------|---------|
| DAU / MAU | 1,000人 / - |
| ピーク時同時アクセス | 約50 RPS |
| データ転送（猫写真） | あり → S3 + CloudFront で配信（ECS負荷・コスト削減） |

### VPC・ネットワーク分離

**VPC 分離方針**: 同一VPC + サブネット分離（stg / prod を同一VPC内で分離。コスト低・シンプル）

| サブネット | 配置コンポーネント |
|-----------|-----------------|
| パブリック | ALB / CloudFront オリジン |
| プライベート | ECS Fargate / RDS |

**環境間通信**: 不要

### IAM 分離戦略

| 対象 | IAMロール / ポリシー方針 |
|------|----------------------|
| ECS タスク | 最小権限ロールを個別作成（S3読み取り専用など） |
| CI/CD（GitHub Actions） | OIDC + 専用ロール（アクセスキー不使用） |
| 開発者（stg 環境） | 書き込み可 |
| 開発者（prod 環境） | 読み取り専用 |

### IaC

**ツール**: Terraform
**state 管理**: S3 + DynamoDB（ロック機能付き）

## リリース頻度と承認フロー

**リリース頻度**: 機能単位（随時）

### 通常リリースフロー

1. feature ブランチで開発・PRを develop へマージ
2. develop → stg 自動デプロイ・動作確認
3. STG 確認後1日以上経過してから まことPM が本番リリース承認
4. develop → main マージ → prod 自動デプロイ
5. リリース完了をメールで通知

**最終承認者**: まことPM
**承認タイミング**: STG 確認後1日以上経過してから

### 緊急リリース（ホットフィックス）フロー

1. main から `hotfix/*` ブランチを作成・修正
2. まことPM が即時承認
3. main と develop へマージ → prod 自動デプロイ
4. リリース完了をメールで通知

**通知先**: メール
