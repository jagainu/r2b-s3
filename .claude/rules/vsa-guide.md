# Vertical Slice Architecture（VSA）ガイド

## 概要

Vertical Slice Architecture は、機能を**縦スライス**として分割し、
各スライスが UI から DB まで一貫した機能を含む設計パターンです。

## VSAの利点

✅ **早期の動作確認**：各スライスが独立して動作可能
✅ **依存関係の明確化**：スライス間の依存が可視化される
✅ **並列開発**：複数スライスを同時に開発可能
✅ **テスト容易性**：各スライスを独立してテスト可能

## スライスの構成

```
Slice: "タスク作成"
├─ Presentation
│  └─ pages/tasks/create.tsx (UI・フォーム)
├─ Business Logic
│  └─ services/CreateTaskService.ts (バリデーション・ルール)
└─ Data Access
   └─ repositories/TaskRepository.ts (DB操作)
```

### 1つのスライスに含まれる責務

| レイヤー | 責務 | 例 |
|---------|------|-----|
| **Presentation** | ユーザーインタフェース、入力受け付け | フォーム、バリデーション表示 |
| **Business Logic** | ビジネスルール、ドメインロジック | タスク優先度の算出、重複チェック |
| **Data Access** | データベース操作、永続化 | INSERT、UPDATE、クエリ |

## 実装順序の決定（依存関係）

```
スライス1: "基本CRUD"（依存なし）
   ↓
スライス2: "タスク検索"（スライス1に依存）
   ↓
スライス3: "AIによる優先度提案"（スライス1, 2に依存）
```

**依存ルール**：
- 上流スライスが完成してから、下流スライスを実装
- 下流スライスのモックを先に作成することで、並列開発も可能

## スライスの粒度

### 粒度が大きすぎる場合
❌ "すべてのタスク管理機能"
→ 実装期間が長く、テストが複雑になる

### 粒度が小さすぎる場合
❌ "フォームのボタンをクリック"
→ スライスの数が多すぎて、管理が煩雑になる

### 適切な粒度
✅ "タスク作成"、"タスク検索"、"タスク削除"
→ 1〜2日で実装可能な機能単位
